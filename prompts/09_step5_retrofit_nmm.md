# Step 5: Retrofit nmm to consume the shared packages

Depends on step 3a (`nmm-events`), step 3b (`nmm-event-contracts`), step 3c (`nmm-tenancy`), step 3d (`nmm-auth`), step 4a (real schemas in `nmm-event-contracts`), step 4b (multi-tenancy contract in `nmm-tenancy`). The dependency on step 4a is hard: `nmm-events`'s `SchemaRegistry` is strict-only and the Delivery API's events must validate against real schemas, not the placeholder.

Read the existing `nmm` repo's `CLAUDE.md` and full `docs/`, then `app/services/outbox/`, `app/workers/`, `app/dependencies.py`, and the existing role-permission and authorisation code wherever it lives in `app/`. Then `nmm_build_plan_v14.md` work order step 5.

## Scope

Move outbox runtime code into `nmm-events`'s usage. Move JWT validation primitive into `nmm-auth`'s usage. Wire the `nmm-tenancy` contextvar shim. Run the existing test suite and the harness to verify nothing breaks.

This step does not move permission-checking code. The role permission matrix, the per-route permission requirements, and the permission service stay in `nmm`. `nmm-auth` is not in the permission-evaluation business.

## Outbox into nmm-events

Replace `app/services/outbox/` and the related worker code with imports from `nmm_events`.

- `app/models/outbox.py`: outbox table now uses `OutboxModel` mixin from `nmm_events.outbox.model`. The mixin was built in step 3a to match the existing nmm table shape, so no destructive migration is needed. If a column on the existing table is not in the mixin, keep it as a Delivery-API-specific extension on the model class. Do not change the table on disk.
- `app/workers/outbox_relay.py`: replaced by a thin entrypoint that constructs `nmm_events.outbox.relay.OutboxRelay` with the Delivery API's RabbitMQ config, exchange name, and routing-key derivation function.
- Outbox writes from services use the mixin's `enqueue` helper rather than raw inserts.
- Dead-letter table uses `nmm_events.dead_letter.DeadLetterEventsModel` similarly. Existing dead-letter rows migrate forward without table modification.

The `SchemaRegistry` runs in strict mode. Application startup constructs it with the production loader: `SchemaRegistry(loader=nmm_event_contracts.loader.load_all_schemas)`. Every event the Delivery API publishes must have a schema at the declared version. If the harness reveals an event without a schema, file the gap as a step 4a follow-up; do not relax to permissive.

## JWT validation into nmm-auth

Replace JWT validation code in `app/dependencies.py` with imports from `nmm_auth`.

- `require_user` is now `nmm_auth.dependencies.require_user("nmm-delivery", {"native_staff"})`.
- The custom claim block parsing uses `nmm_auth.claims.parse_claims`.
- The Auth0 Action endpoint at `GET /internal/users/{auth0_user_id}/claims` uses `nmm_auth.dependencies.require_auth0_action_secret`.
- `require_internal_service("nmm-internal")` is added for the outbox replay endpoint per v16 section 9. Add the endpoint `GET /internal/outbox/events` if it does not already exist.

The existing role-permission code stays in `nmm`. Permission checks per route, the role-permissions admin tables, the permission service: all unchanged in this PR. They consume `CurrentUser` from `nmm-auth`'s `require_user` rather than from the previous local equivalent.

## Tenancy contextvar shim

The Delivery API does not enforce tenancy (Native staff bypass is the only mode), but it depends on `nmm-tenancy` for the contextvar shim that the `nmm-events` consumer worker uses to set tenant identity on outbound events.

- `require_user` for the Delivery API sets `nmm_tenancy.contextvar.set_native_bypass()` on every authenticated request.
- The outbox relay reads the contextvar where it's already set in process; for events generated by RQ workers without an HTTP request, the worker explicitly sets `set_native_bypass()` on each job.

No SQLAlchemy listener registration on the Delivery API: there are no tenant-scoped tables here.
