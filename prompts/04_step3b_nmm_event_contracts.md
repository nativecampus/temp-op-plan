# Step 3b: nmm-event-contracts

Read v16 section 8 (Cross-API event catalogue), `nmm_build_plan_v14.md` section "`nmm-event-contracts` repo" and the work order step 3.

## Scope

`nmm-event-contracts` is the JSON Schema repo. It publishes a Python package and a TypeScript types package on every tagged release. The Python package is consumed by the three APIs at runtime via `nmm-events`'s `SchemaRegistry`. The TypeScript types package is consumed by the three SPAs.

This step ships the package infrastructure: directory layout, registry format, build pipeline, Python and TypeScript packaging, CI. A single placeholder schema is included so the build pipeline has something to operate on. The actual schemas for v16 events are step 4a.

## Repo layout

```
schemas/
  <api>/<aggregate>/<event_type>/v<n>.json
  _shared/                      # shared $defs for entity shapes
registry.json                    # routing key -> current version, publishers, consumers, action

py/
  pyproject.toml                 # nmm-event-contracts Python package
  src/nmm_event_contracts/
    __init__.py
    loader.py                    # exposes load_all_schemas() -> dict[routing_key, dict[version, schema]]
    registry.py                  # exposes load_registry() -> registry data

ts/
  package.json                   # @native/nmm-event-contracts-types
  src/                           # generated by json-schema-to-typescript at build
```

## Schema directory layout

Routing key `delivery.campaign_booking.state_changed` maps to `schemas/delivery/campaign_booking/state_changed/v1.json`.

A version is a separate file. v1 stays as `v1.json` when v2 lands as `v2.json`. The `SchemaRegistry` in `nmm-events` consumes whichever versions are present.

Cross-schema references via `$ref` to a shared `_shared/` directory are acceptable for entity shapes referenced by multiple events. Example: `_shared/booking.json` referenced from `delivery.campaign_booking.created` and `delivery.campaign_booking.updated`.

## Placeholder schema

Single file at `schemas/delivery/_placeholder/created/v1.json`. JSON Schema 2020-12. Standard event envelope (`event_id`, `event_type`, `aggregate_type`, `aggregate_id`, `schema_version`, `occurred_at`, `published_at`, `publisher`) plus an empty `payload` object.

## registry.json

Maps each routing key to:

- `current_schema_version`: integer
- `publisher_api`: one of `delivery`, `su_mm`, `advertiser`
- `consumer_apis`: array
- `consumer_action`: human-readable description of what each consumer does on receipt

One placeholder entry covering the placeholder schema:

```json
{
  "routing_key": "delivery._placeholder.created",
  "current_schema_version": 1,
  "publisher_api": "delivery",
  "consumer_apis": [],
  "consumer_action": "Placeholder for build pipeline; replace in step 4a"
}
```

## Python package

`nmm_event_contracts.loader.load_all_schemas()` walks the `schemas/` directory and returns a dict keyed by routing key, with each value being a dict from version integer to the parsed JSON schema document. `nmm-events`'s `SchemaRegistry` calls this at startup.

`nmm_event_contracts.registry.load_registry()` parses `registry.json` and returns it as a dict.

The package is a thin wrapper. The schemas themselves are the data; the Python module just loads them.

## TypeScript package

`@native/nmm-event-contracts-types` generates TypeScript types from `schemas/` using `json-schema-to-typescript` on every build. Exports per-routing-key types named after the routing key in PascalCase (e.g. `DeliveryCampaignBookingStateChangedV1`).

Build runs as part of CI on every PR and as part of the release pipeline on every tag.

## CI

On every PR:

- JSON Schema validation on every file under `schemas/` (catches malformed JSON Schema)
- Consistency check: every routing key in `registry.json` has a file at the declared `current_schema_version`, every file under `schemas/` (excluding `_shared/`) has a registry entry
- Python package build and unit tests
- TypeScript package build (verifies types generate cleanly from every schema)

A version bump is required on any change to `schemas/` or `registry.json`. CI enforces by comparing `package.json` and `pyproject.toml` versions against `main`. The two packages version in lockstep.

## Publishing

Both packages publish to the private package registry on every tagged release. Tag format `v<semver>`. The Python package is consumed by `nmm-events`. The TypeScript package is consumed by application repos' SPAs that need typed event payloads.

## Out of scope

The schemas for v16 events. That is step 4a. This step ships only the placeholder schema and the package infrastructure.

The runtime code that loads and validates against these schemas. That is in `nmm-events` (step 3a).

## Acceptance

Tests pass. The Python package builds and publishes a 0.1.0 release containing the placeholder schema. The TypeScript package builds and publishes a 0.1.0 release with one generated type. `nmm-events`'s `SchemaRegistry` (from step 3a) loads the package successfully at startup and validates a sample payload against the placeholder schema in strict mode.
